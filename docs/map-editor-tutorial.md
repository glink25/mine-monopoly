## 0. 开场 & 项目概览

MineMonopoly 是一款多人在线大富翁游戏。地图编辑器是其中独立运行的桌面应用（基于 Electron），用于可视化地创建和编辑游戏地图。

### 你能用它做什么

- 在 3D 场景中摆放地图格子
- 配置可购买、可升级的地产
- 编写事件和机会卡的游戏逻辑
- 设计玩家角色和初始化属性
- 自定义游戏流程（掷骰子、移动、回合结算等）
- 导出地图并在客户端中运行

## 2. 资源导入 — 3D 模型

在构建地图之前，先编写地图基本信息，再准备好视觉资源。

### 2.1 地图信息

打开顶部工具栏的「地图属性」，填写以下内容：

| 字段 | 说明 |
|------|------|
| 地图名称 | 地图的显示名称（必填，否则编辑器报错） |
| 作者名称 | 地图制作者署名 |
| 地图版本 | 版本号，如 `1.0.0` |
| 地图说明 | 地图的文字介绍 |
| 地图背景 | 封面/缩略图用的背景图片（建议，本地上传） |
| 地图封面 | 地图选择列表中展示的封面图片（建议，本地上传） |

### 2.2 模型管理

打开顶部工具栏的「模型管理」弹窗。

- **支持的格式**：目前仅支持 `.glb` / `.gltf`
- **单个导入**：点击「添加模型」，在弹窗中选择文件，导入前可通过 Transformation Panel（位置/旋转/缩放滑块）**预览并调整模型姿态**，确认后点击「添加」烘焙变换并保存
- **批量导入**：点击「批量添加模型」，多选文件后自动逐个导入，每个模型自动以**文件名作为名称**
- **预览**：管理器中每个模型卡片内嵌 3D 渲染器，可直接旋转查看
- **删除**：删除模型会触发**级联清理**——自动删除引用该模型的所有 MapItemType，进而删除所有基于这些类型的 MapItem；同时将 `buildingModelIdList` 中引用该模型的位置置空

### 2.3 默认建筑模型

当地皮升级时，需要不同等级的建筑模型。在「默认房屋模型」弹窗中配置：

| 等级 | 说明 |
|------|------|
| 0 级 | 空地（无建筑） |
| 1 级 | 初级建筑 |
| 2 级 | 中级建筑 |

每级可以指定不同的 3D 模型，玩家建造时自动替换。

## 3. 搭建地图 — 地图项与路径

这是地图制作的核心环节 — 在地面上摆放格子并设定行走路线。

### 3.1 地图项类型（MapItemType）

`MapItemType` 是地图项的**分类标签**，不包含任何游戏功能逻辑。它仅用于：

- **视觉归类**：通过颜色和名称区分不同类型的地图格子
- **3D 模型绑定**：每个类型绑定一个模型，该类型下的所有 MapItem 都会渲染为该模型
- **路径索引选项**：生成 MapIndex 时可勾选要包含的类型

> MapItem（地图格子）的功能由自身的属性决定：绑定 `property` 即为地产格子、绑定 `mapEventId` 即触发事件。类型仅是一个方便管理的标签，不决定功能。

创建类型时常见的命名参考：

| 命名示例 | 说明 |
|----------|------|
| 起点 | 通常放在地图索引第一位，游戏中经过时触发相关逻辑 |
| 地产 | 通常对应绑定了 `property` 属性的格子 |
| 事件点 | 通常对应绑定了 `mapEventId` 的格子 |
| 特殊区域 | 自定义命名的其他格子 |

> 这些名称仅作建议，不影响任何游戏逻辑。`MapItemType` 的实际字段仅包含：`id`、`color`、`name`、`modelId`、`size`。

**创建步骤**：
1. 在左侧面板选择「创造模式」
2. 点击「新增类型」，在弹出的表单中：
   - **选择模型**（必选）— 从已导入的模型下拉列表中选择，选择后名称自动填入模型文件名
   - **填写名称**（必填）— 默认自动填入模型名称，类型颜色随机生成
3. 创建完成后在类型列表中会显示颜色方块和名称
4. 点击类型卡片将其设为当前选中（3D 画布中会显示该模型的预览），在地图上点击即可放置
5. **查看所有类型**：点击「查看所有类型」按钮可查看全部已创建类型的表格（含颜色、名称、ID）

> 放置时可使用快捷键：`R` 旋转当前选中物块，`Q` 取消选中

### 3.2 放置与编辑

- **放置**：在创造模式下选中类型后，在 3D 场景中点击即可放置
- **选择模式操作**：切换到选择模式，点击已放置的地图项
  - **微调位置**：点击左面板的方向控制板（上/下/左/右），每步移动 1 格
  - **旋转**：逆时针 90° / 顺时针 90° 按钮
  - **绑定地产**：点击「绑定地皮」→ 进入连线模式 → 点击另一个地图项完成绑定
  - **解除绑定**：已绑定时可点击「解除绑定」
- **删除**：选中后按 `Delete` / `Backspace` 键
- **撤销删除**：`Ctrl+Z` 撤销最近一次删除操作（支持多级撤销）

### 3.3 路径索引（MapIndex）

路径索引告诉游戏角色按什么顺序走格子。

**配置步骤**：
1. 打开「路径索引编辑器」
2. 在「选择类型用来生成索引」多选框中勾选需要纳入路径的**地图项类型**（MapItemType）——系统会筛选出这些类型下的所有 MapItem
3. 可选：输入起点 MapItem 的 ID（不填则使用筛选结果中的第一个）
4. 点击「生成路线」

**生成原理**：以选中的 MapItemType 过滤出目标 MapItem，然后对这些格子的 (x, y) 坐标进行 **DFS（深度优先搜索）四方向遍历**（上/下/左/右），自动推导出合理的行走顺序。结果存入 `mapIndex`（MapItem ID 数组），游戏运行时玩家将按此顺序行走。

### 3.4 批量操作

在俯视视角下启用框选模式，可以框选多个地图项，然后：

- 批量移动（方向键）
- 批量旋转
- 批量删除

## 4. 地产系统

让地图格子变成可购买、可升级的资产。

### 4.1 绑定地产

绑定地产本质是建立 **「入口格 → 地产格」** 的映射关系，分两步操作：

1. 在「选择模式」下选中一个地图项（作为**入口格**），点击左侧面板的「绑定地皮」→ 进入连线模式
2. 在 3D 场景中**点击另一个地图项**（作为**地产格**）完成绑定

**绑定后的效果**：

```
入口格 (source)                  地产格 (target)
  linkto ──────────→              beLinked ← 记录是谁绑定了自己
  (玩家走到这里)                   property → 承载价格、过路费、建筑模型
```

- **入口格**：玩家走到的格子，游戏检测到 `linkto` 后跳转到 target 触发地产逻辑
- **地产格**：承载 `property` 数据（价格、过路费等），建筑模型也渲染在这个格子的位置上
- **建筑朝向**：渲染时建筑会自动朝向入口格的方向

这种设计允许入口和建筑在空间上分离——例如一个入口格对应一个单独的大型建筑格子。

### 4.2 基础参数

| 参数 | 说明 | 示例 |
|------|------|------|
| 名称 | 地产名称 | "天空之城" |
| 购买价格 | 玩家购买需支付的金额 | 200 |
| 升级费用 | 每升一级的费用 | 100 |
| 最大等级 | 最高可升到几级 | 3 |
| 过路费 | 每级对应的过路费数组 | [50, 100, 200, 400] |

### 4.3 地产编辑模式

地产支持 **标准** 和 **自定义** 两种编辑模式，通过开关切换：

- **标准模式**：配置购买价格、升级成本、各级过路费。`maxLevel` 自动与过路费等级数同步
- **自定义模式**：手动编辑最大等级，并可编写自定义 effectCode 脚本（通过「编辑脚本代码」按钮打开弹窗）

### 4.4 效果代码（effectCode）

使用效果代码编辑器（基于 Monaco Editor）为地产添加初始化逻辑。地产的 effectCode 是**同步函数**，在游戏初始化时对每块地产执行一次（非每次玩家到达时执行）。

**基础示例** — 地产的 effectCode 参数为地产属性和游戏进程：

```typescript
((property: IProperty, gameProcess: IGameProcess) => {
  // 在游戏初始化时对地产进行自定义设置
  // property 包含地产的名称、等级、持有者等信息
  // 基础过路费收取由游戏引擎自动处理
  // 也可以通过修饰器系统给地产添加修饰器，处理玩家到达该地产的事件
})
```

**代码编辑器使用技巧**：
- **自动补全**：TypeScript IntelliSense 在输入时自动触发，提供参数类型提示和代码补全
- **类型提示**：悬停查看类型
- **校验**：编辑器内置 TypeScript 校验，错误会标红

### 4.5 建筑模型与自定义数据

- **建筑模型绑定**：点击「选择模型」打开弹窗，为每级地产指定对应的 3D 建筑模型（玩家建造时自动替换）
- **自定义 UI**：从下拉列表选择 UI 模板，绑定后地产信息面板会使用该模板渲染（地产实例 `property` 会传入作为数据）
- **自定义数据（Custom Data）**：动态键值对编辑器，支持 string / number / boolean 三种类型，可增删行

## 5. 地图事件 & 机会卡

让地图上的格子产生各种效果。

### 5.1 地图事件

地图事件通过设置**类型**来控制触发方式：

| 类型 | 触发时机 |
|------|----------|
| 到达事件（ArrivedEvent） | 玩家停在该格子上时触发 |
| 经过事件（PassedEvent） | 玩家经过该格子时触发（不停留） |
| 普通事件（NormalEvents） | 自定义触发条件 |

**创建步骤**：
1. 打开「地图事件管理器」
2. 点击新增，填写：
   - **名称**（必填）、**描述**（必填 textarea）
   - **类型**（必选）：到达事件 / 经过事件 / 普通事件
   - **图标**（必选）：从本地上传图片
   - **effectCode**：编写地图事件效果代码
3. 保存事件后，回到选择模式，选中目标地图项，在左侧面板的「地图事件选择器」下拉选中该事件，点击「绑定」即可

> 编辑事件时，选择图标会自动保存为新的图片资源

**常见效果示例**：

扣钱：
```typescript
(async (player: IPlayer, gameProcess: IGameProcess) => {
  // 玩家支付 50 元给银行
  await player.cost(50);
})
```

加钱：
```typescript
(async (player: IPlayer, gameProcess: IGameProcess) => {
  // 玩家获得 200 元
  await player.gain(200);
})
```

传送到起点：
```typescript
(async (player: IPlayer, gameProcess: IGameProcess) => {
  // 传送玩家到索引为 0 的位置（起点）
  await player.tp(0);
})
```

编写完成后，将事件绑定到对应的地图项上。

### 5.2 机会卡

机会卡是由游戏运行时的自定义代码（如地图事件、阶段逻辑、商店系统等）发放给玩家的。开发者可以在 effectCode 中通过游戏进程 API 控制机会卡的发放时机和对象。

**类型**：

| 类型 | 说明 |
|------|------|
| ToSelf | 仅对抽卡者生效 |
| ToOtherPlayer | 对其他玩家生效 |
| ToPlayer | 对指定玩家生效 |
| ToProperty | 与地产相关 |
| ToMapItem | 与地图项相关 |

**创建步骤**：
1. 打开「机会卡管理器」
2. 点击新增，填写：
   - **名称**（必填）、**描述**（必填 textarea）
   - **类型**（必选）：选择目标类型（ToSelf / ToOtherPlayer 等），切换类型时 effectCode 模板**自动更新参数签名**
   - **颜色**：使用取色器设置卡牌颜色
   - **图标**（必选）：从本地上传图片
   - **effectCode**：编写效果代码
3. 编辑时右侧可**实时预览卡牌样式**
4. 保存后可在代码中通过 `createNewChanceCard(cardId)` 引用

**示例 — 给玩家发放机会卡（如在商店中购买）**：
```typescript
(async (sourcePlayer: IPlayer, target: IPlayer, gameProcess: IGameProcess) => {
  // 从机会卡模板创建一张新卡（参数为模板 ID）
  const card = gameProcess.createNewChanceCard('chance_card_lucky');
  // 将机会卡发放给玩家
  await sourcePlayer.gainCard(card);
})
```

**示例 — 对自己生效（ToSelf，加钱）**：
```typescript
(async (sourcePlayer: IPlayer, target: IPlayer, gameProcess: IGameProcess) => {
  // 抽卡者获得 200 元
  await sourcePlayer.gain(200);
})
```

**示例 — 对其他玩家生效（ToOtherPlayer，收税）**：
```typescript
(async (sourcePlayer: IPlayer, target: IPlayer, gameProcess: IGameProcess) => {
  // 让目标玩家支付 50 元
  await target.cost(50);
})
```

### 5.3 代码校验

所有 effectCode 在保存前会自动进行 TypeScript 类型校验。

```

```

## 7. 游戏流程定制 — 阶段系统

阶段系统控制游戏的完整流程。阶段在 GameProcessWorker 中按顺序依次执行，每个阶段是一个独立的 `initEventCode` 函数。

> 💡 编辑器自带的系统阶段模板（如掷骰子、移动、到达处理等）已经实现了最基本的游戏流程，**无需任何修改即可运行**。开发者只需在此基础上按需定制即可。

### 阶段执行流程

Worker 的完整执行顺序如下：

```
启动游戏
├─ playerPreInit       — 玩家预初始化（先于角色 initCode）
├─ propertyPreInit     — 地皮预初始化（先于地皮自定义逻辑）
├─ gameInited          — 游戏初始化完成
├─ [如有存档] postRestore — 存档恢复后执行
│
└─ 主循环 gameLoop()
   ├─ gameRoundStart   — 每回合开始阶段
   ├─ 遍历每位玩家 → playerRound（玩家回合，可含多段连续阶段）
   └─ gameRoundEnd     — 每回合结束阶段
```

**关键理解**：`playerRound` 类别下可以定义多个连续阶段，例如 `rollDice → playerMove → arrivedEvent → playerRoundEnd`，它们依次按数组顺序执行。这些阶段之间**没有硬编码的流程控制**，每个阶段的逻辑完全由编辑器中的 `initEventCode` 决定。

例如一个自定义的 playerRound 阶段数组：

```
playerRound 阶段列表（编辑器中定义的顺序）
  ├─ rollDice           — 掷骰子（编写掷骰逻辑）
  ├─ playerMove         — 移动（编写移动逻辑）
  ├─ arrivedEvent       — 到达处理（处理购买/过路费）
  └─ playerRoundEnd     — 回合结束（清理/结算）
```

### 阶段分类

| 类别 | 执行时机 | 说明 |
|------|----------|------|
| `playerPreInit` | 启动时 | 玩家初始化之前执行，可用于注入自定义玩家数据 |
| `propertyPreInit` | 启动时 | 地皮初始化之前执行 |
| `gameInited` | 启动时 | 玩家和地皮初始化完成后执行，一次性 |
| `postRestore` | 启动时 | 仅在加载存档数据后执行 |
| `gameRoundStart` | 每轮开始 | 所有玩家回合开始前执行 |
| `playerRound` | 每位玩家 | **玩家回合内的阶段序列**，可包含多个子阶段 |
| `gameRoundEnd` | 每轮结束 | 所有玩家回合结束后执行 |

> `gameOverRule` 并非独立执行的阶段——它是地图数据中一段特殊的初始化代码，编译为一个结束判定函数。Worker 在**每次阶段执行完毕后**（包括回合开始、玩家回合内每个子阶段、回合结束）都会自动调用此函数检查游戏是否该结束。

### 阶段编辑器使用

流程管理器左侧是树形阶段列表，分为 8 个组：

```
├─ 游戏结束判定规则     ← 返回 false=继续 / 玩家ID数组（排名）
├─ 玩家预初始化         ← 玩家初始化前运行
├─ 地皮预初始化         ← 地皮初始化前运行
├─ 游戏基础内容初始化后  ← 一次性
├─ 存档恢复后           ← 仅读取存档时执行
├─ 轮次循环区域（橙色框）
│  ├─ 轮次开始阶段
│  ├─ 玩家回合阶段（蓝色框）← 系统默认含：回合开始→掷骰→移动→到达→回合结束
│  └─ 轮次结束阶段
```

1. 点击各阶段之间的 **「+」按钮** 可在该位置添加新阶段
2. **系统阶段**（带 `mark` 标签或来源为「系统」）**不可删除**
3. 用户自定义的阶段可删除
4. 选中阶段后，右侧代码编辑器编写其 `initEventCode`

**系统默认的 playerRound 模板（伪代码）**：

编辑器自带的系统模板已覆盖完整的回合流程，无需修改即可运行：

| 阶段 | 默认模板逻辑（伪代码） |
|------|----------------------|
| 掷骰子 | 等待玩家操作 → 允许使用机会卡 → `player.rollDices()` → 将 `diceResult` 写入 context |
| 移动 | `player.walk(骰子总点数)` → 将最终 `positionIndex` 写入 context |
| 到达 | `gameProcess.handleArriveEvent(player)` → 自动处理购买/交租/事件触发 |
| 回合结束 | 清理当前回合数据 |

**实际模板示例**（以移动阶段为例）：

```typescript
(async (context, gameProcess: IGameProcess) => {
  const player = context.currentRoundPlayer;
  await player.walk(context.diceResult.map(d => d.result).reduce((p, c) => p + c, 0));
  context.targetIndex = player.positionIndex;
}) as GameEventFunction<PlayerMoveContext>;
```

## 8. 修饰器 & UI 模板 & 额外库

三个高级系统，让地图功能更丰富。

### 8.1 修饰器模板

修饰器（Modifier）可以在特定时机**临时修改或监听**游戏命令的行为。每个修饰器在游戏中生成独立实例。

**应用场景**：
- 「经过某地时双倍过路费」（持续 3 回合）
- 「免单一次」（消耗型）
- 「每回合开始额外获得 100 元」（持续型）

**创建步骤**：
1. 打开「修饰器模板管理器」
2. 在表单左侧配置描述符，右侧编写 effectCode。切换 `commandType` 时，代码编辑器的**函数头部会自动更新**（参数个数、类型和泛型签名随命令类型变化）

| 字段 | 说明 |
|------|------|
| `timing` | 触发时机：`"before"`（命令执行前拦截）/ `"after"`（命令执行后回调） |
| `commandType` | 监听的命令类型，如 `player.money.gain`、`property.arrived` 等 |
| `remainingTriggers` | 剩余触发次数（`-1` 表示无限触发） |
| `priority` | 优先级，数字越大越先执行（默认 `0`） |
| `autoConsume` | 触发后是否自动消耗次数（默认 `true`） |


3. 编写 effectCode（异步函数，4 个参数）：

```typescript
(async (player: IPlayer, gameProcess: IGameProcess,
  cmd: ICommand<PlayerCommandMap, "player.money.gain">,
  ctx: ICommandContext<PlayerCommandMap, "player.money.gain">) => {
  // 双倍金额：将命令中的金额翻倍
  cmd.payload.money = (cmd.payload.money ?? 0) * 2;
  return cmd;
})
```

> 第1个参数名和类型由 commandType 前缀决定：`player.*` → `player: IPlayer`；`property.*` → `property: IProperty`
>
> `ctx` 上下文提供以下控制方法：
> - `ctx.cancel(result)` — 取消命令执行并返回指定结果
> - `ctx.setResult(result)` — 设置命令的返回值
> - `ctx.skip()` — 跳过本次触发次数消耗
> - `ctx.consume(count)` — 自定义消耗次数

4. **分类管理**：修饰器模板管理器支持按 **来源（source）** 筛选、**标签（tag）** 筛选和**搜索框**搜索（匹配 name / slug / commandType），模板会按来源分组显示
5. 编辑时可填写 **Meta 信息**：显示名称、描述、来源、触发时机名称、标签
6. 点击「复制标识」可复制 `$mod__{slug}` 供代码中引用
7. 保存模板，游戏中通过 `modifierManager.add()` 添加实例

### 8.2 UI 模板

UISchema 是一种声明式的 UI 描述方式，由 UiRenderer 统一渲染。UI 模板管理器分为**两个标签页**：

- **组件库（Library）** — 管理 UI 模板定义
- **自定义UI（Map）** — 管理地图上的 UI 实例

**创建模板（组件库）**：
1. 打开「UI 模板管理器」→ 组件库标签页
2. 新建模板，填写：
   - **名称**、**slug**（用于代码中引用 `$ui__{slug}`）
   - **template**：树形 JSON Schema 编辑器，定义 UI 结构和样式
3. 保存后点击「复制UI标识」可复制 `$ui__{slug}`
4. 删除模板时若已被地图上的实例引用，则**禁止删除**

**UISchema 示例**：

```json
{
  "type": "panel",
  "children": [
    { "type": "text", "value": "地产名称" },
    { "type": "text", "value": "等级: {property.level}" },
    { "type": "text", "value": "持有者: {property.owner}" }
  ]
}
```

**创建实例**：
1. 在「CustomUI 管理器」中创建实例
2. 选择引用的 UI 模板
3. 设置布局（x, y, width, height）
4. 实例会在游戏中对应位置渲染

### 8.3 额外类型库（ExtraLibs）

ExtraLibs 是全局共享的 TypeScript **类型声明**，所有 `effectCode` / `initCode` / `initEventCode` 的代码编辑器都能使用其中的类型。它以 `.d.ts` 声明文件的形式注入 Monaco Editor，**不会在游戏运行时执行**。

**用途**：
- 定义共享类型（interface / type）
- 声明全局常量类型
- 编写辅助类型的泛型工具

**示例**：

```typescript
// 声明全局类型
type MoneyTagType = 'card' | 'event' | 'property' | 'shop';

// 声明常量类型
declare const PASS_GO_BONUS = 200;
declare const TAX_RATE = 0.1;

// 声明业务类型的形状
interface RoleExtraData {
  title: string;
  specialSkill: string;
}
```

保存后，所有 effectCode 中即可直接使用这些类型进行类型标注。业务逻辑代码应写在对应的 `effectCode` / `initEventCode` 中，而非 ExtraLibs。

## 9. 游戏参数 & 导出测试

### 9.1 游戏设置

游戏参数采用**动态表单**机制，内置字段不可删除，用户可自由添加自定义参数。

**操作方式**：
- **添加字段**：点击「添加数字输入」或「添加下拉选择」按钮
- **编辑字段**：每个字段可配置：
  - **Label**（显示名称）
  - **Key**（字段标识，代码中通过 `GameSetting.key` 访问）
  - **DefaultValue**（默认值）
  - **数字输入**额外：Min（最小值）、Max（最大值）
  - **下拉选择**额外：选项列表（显示名 + 值，可动态添加/删除）
- **删除字段**：非内置字段可删除

### 9.2 导出地图

工具栏 → 文件 → 导出，支持两种格式：

| 格式 | 说明 | 用途 |
|------|------|------|
| `.fpmap` | 地图源文件，包含完整原始数据 | **编辑器和客户端均可使用**，适合继续编辑和分享 |
| `.mmmap` | 压缩构建产物，打包后体积更小 | **仅客户端使用**，适合发布和线上部署 |

### 9.3 在客户端中测试

1. 将导出的 `.fpmap` 文件放入客户端项目的地图目录
2. 启动游戏客户端：`pnpm dev-client`
3. 创建房间，选择刚制作的地图
4. 加入游戏，实际跑一圈验证效果

### 9.4 调试技巧

- effectCode 中的错误会输出到浏览器控制台
- 使用 `console.log` 输出临时调试信息（在浏览器 DevTools 中查看）
- 代码编辑器中的 TypeScript 校验可以提前发现类型错误

## 10. 收尾总结

### 📋 地图发布检查清单

编辑器会自动检测地图的完整性，在界面左下角以 **Error（红色）** 和 **Warning（黄色）** 两种级别展示。以下是导出地图前必须满足的条件：

**🔴 Error — 必须满足（缺少则地图无法正常运行）：**

| 检查项 | 说明 | 对应章节 |
|--------|------|----------|
| 空的地图 | 至少放置一个地图项 | §3 |
| 没有设置地图名称 | 在地图属性中填写名称 | §2.1 |
| 没有设置地图索引路径 | 配置 MapIndex 行走路径 | §3.3 |
| 没有角色 | 至少创建一个角色 | §6 |
| 没有加入模型 | 至少导入一个 3D 模型 | §2.2 |
| 没有设置地皮等级(房屋)模型 | 配置 0~2 级的默认建筑模型 | §2.3 |
| 被绑定地皮没有设置地皮参数 | 已绑定地产的地图项必须填写价格、过路费等参数 | §4.2 |

**🟡 Warning — 建议完善（缺少不影响运行但体验不完整）：**

| 检查项 | 说明 |
|--------|------|
| 没有设置地图背景 | 建议设置背景图片 |
| 没有设置地图封面 | 建议设置封面图片 |
| 没有机会卡 | 建议创建至少一张机会卡 |

> 💡 编辑器的 **alertList** 会实时检测以上所有项，在界面上展示当前缺失项。导出地图前请确保无 Error 级别提示。

### 完整制作流程回顾

```
资源准备（3D 模型）
    ↓
搭建地图（地图项 + 路径索引）
    ↓
配置地产（绑定 + 参数 + effectCode）
    ↓
编写逻辑（地图事件 / 机会卡 / 阶段）
    ↓
设计角色（属性 + initCode）
    ↓
高级定制（修饰器 / UI 模板 / ExtraLibs）
    ↓
配置参数 + 导出测试
    ↓
✅ 完成！
```
